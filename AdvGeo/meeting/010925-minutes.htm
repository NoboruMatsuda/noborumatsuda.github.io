Issues around "completeness"

Definition of Construction

"Construction" could be defined either by points or by segments constructed.

If it's point that determines a construction, then two constructions with the same set of points but differ in the segments connecting them would be considered as identical. 

"Constructions" with the same configuration could be treated as either the same or the different constructions depending on the order of points (hence segments) constructed.

"Constructions" with the same configuration drawn in the same way could be treated as the same or the different constructions depending on the constraints (or assertions if you will) associated with the construction. 

For example, draw segment AB as a perpendicular to XY to prove a goal G by showing AB // EF

vs.

draw segment AB to be parallel to EF to prove a goal G by showing AB is a perpendicular to XY

"Redundancy" of construction

Is it true that GRAMY can find all the proofs with a construction such that all the element (could be only points) are referred in a proof?

Or, can you make an arbitrarily complicated (or redundant) construction?  -- Don't think about constructions with transitive substitutions, because GRAMY can deal with them.  

Object of Completeness

All proofs -- GRAMY could be "complete" in a sense that it can generate all the possible proofs.
Same proofs -- GRAMY could be "complete" in a sense that it can generate all the "equivalent" proofs, which consider two constructions with different order and/or different constraints as the same constructions.  
Proof all -- GRAMY could be "complete" in a sense that it can proof all the true theorems.

Issues around an ITS design

Definition vs. Assertion

To construct a new segment or a new point point is to define new object, which is assessed as well-defined or not (they can't be wrong or correct).  

A proof is composed by a set of assertions inputted, which you can tell true or false.

Relevancy of a statement

It is actually an interesting research questions that "do you need to allow students to go down a garden path?"  

They would certainly learn something from an impasse.  

Educational objectives

We could teach students geometric concepts, which the teachers claim what they mainly do.

We could teach students problem solving (i.e., search) skills for geometry proof, which is what the teachers actually do. 

Local heuristics vs. abstract plans

To teach problem solving skills (i.e., search), you can either provide students with (1) the local heuristics that tells you which operator to take in what situation, or (2) the abstract plans that tells you how to assemble a solution.  

Andes-III is intended to provide abstract planning for students to learn.  

Remember the capillary graph that represent a planning space in the top half and an instantiations of the plan in the bottom half. 

Model tracing with an "incomplete" solution graph

We are working on a domain with huge solution space that is hardly captured as a solution graph.
